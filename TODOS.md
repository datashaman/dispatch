### P2: Config Editor — config diff preview on save
**What:** Show a side-by-side diff modal (old YAML vs new YAML) before writing dispatch.yml to disk. User confirms or cancels.
**Why:** Builds trust for users editing complex configs — they can see exactly what changed before committing to disk. Especially valuable when editing YAML directly.
**Pros:** Trust-building UX. Catches accidental changes. Uses existing YAML dump for both sides.
**Cons:** Adds an extra click to every save. May feel heavy for simple edits.
**Context:** The config editor already tracks `$originalConfigData` (loaded from disk) and `$configData` (current form state). Diff can be generated by `Yaml::dump()` on both arrays and using a PHP diff library or simple line-by-line comparison rendered with Tailwind classes (green for additions, red for removals).
**Depends on:** Config editor save flow (feat/config-editor branch).

### P2: Config Editor — "Test this rule" simulation
**What:** Button on each rule card that simulates a webhook event against that rule using `RuleMatchingEngine`. Shows "Would match" / "Would not match" with filter evaluation details.
**Why:** Users currently have no way to verify their filters work until a real webhook arrives. This closes the feedback loop.
**Pros:** Instant feedback on rule filters. Reduces debugging time. Uses existing `RuleMatchingEngine::match()`.
**Cons:** Needs a way to input/select test event payloads — adds UI complexity (textarea for JSON? dropdown of recent webhooks?).
**Context:** `RuleMatchingEngine::match()` already evaluates filters against event data. The simulation would call this with a mock event payload and display the result. Could also pull recent webhook payloads from the database as test data.
**Depends on:** Config editor rule cards (feat/config-editor branch). Would benefit from webhook log integration.

### P3: Config Editor — beforeunload warning for unsaved changes
**What:** Add browser `beforeunload` warning when navigating away from the config editor with unsaved changes.
**Why:** Prevents accidental data loss when a user edits rules but navigates away before saving. Standard UX pattern for editors.
**Pros:** Cheap to implement (Alpine.js `x-on:beforeunload` handler + dirty state tracking already planned). High-trust UX signal.
**Cons:** None significant. Minor annoyance if user intentionally abandons edits.
**Context:** The config editor tracks dirty state via a `$isDirty` Livewire property (comparing `$configData` to `$originalConfigData`). The `beforeunload` handler just needs to check this flag. Could be an Alpine directive on the page wrapper.
**Depends on:** Config editor form editing (feat/config-editor branch).

### P2: Event Sources Beyond GitHub
**What:** Abstract the webhook endpoint to support GitLab, Bitbucket, Slack, Linear, Sentry, PagerDuty, and any webhook-sending service. Each source gets its own signature verification, payload normalization, and event type mapping.
